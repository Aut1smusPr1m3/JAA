### Three-stage pipeline summary
| Stage | Module | Always? | What it does |
|-------|--------|---------|--------------|
| 1 | `Ultra_Optimizer.py` | ✅ | Kinematic opt, M204 injection, window detection |
| 2 | `GCodeZAA/gcodezaa/process.py` | ⚙️ optional | Surface-aware Z-adjust, non-planar ironing, STL raycasting |
| 3 | `Ultra_Optimizer.py` via `ArcWelder.exe` | ⚙️ optional | Arc compression (G1→G2/G3) |

### Key safety invariants
- Smoothing hard-capped at **≤ 45°** (`config.py`).
- Build-plate floor enforced: **Z ≥ 0.0** (sanitize + fail pass).
- Machine start/end G-code is **always preserved unchanged**.

## 3  Recent Changes (Cycle 1 → Cycle 2)

### Stage 2 transform observability (committed 2026-04-13)
- **`Ultra_Optimizer.py`** — Added `OBJECT_POSITION_HINT_RE`, `OBJECT_ROTATION_HINT_RE`, and three new helpers:
  - `_parse_exclude_object_define_args` — tokenises `EXCLUDE_OBJECT_DEFINE` key=value lines.
  - `_extract_transform_hints_from_gcode` — finds position/rotation hints from both comment-style and slicer-emitted `EXCLUDE_OBJECT_DEFINE` metadata.
  - `_infer_xy_center_from_gcode_window` — infers object XY center from printable-window motion while honouring G90/G91/G92 state.
  - `resolve_stage2_object_transform` — unified resolver; priority: explicit hint > inferred bounds > origin (0,0); returns dict with `center_x/y`, `rotation_deg`, `source`, window indices, and `inferred_bounds`.
- **`GCodeZAA/gcodezaa/process.py`** — `load_object` now accepts `rotation_deg`; applies an in-plane rotation matrix about the mesh centroid via Open3D tensors; recalculates bounds after rotation.
- **`GCodeZAA/gcodezaa/__main__.py`** — `--position` now accepts `x,y` or `x,y,rotation_deg`.
- **`docs/01-user-guides/orcaslicer-integration.md`** — Documents `ZAA_OBJECT_POSITION`, `ZAA_OBJECT_ROTATION_DEG`, and `EXCLUDE_OBJECT_DEFINE` hint formats.
- **`docs/01-user-guides/troubleshooting.md`** — Section added for transform alignment issues.
- **`test_ultra_optimizer_stage2.py`** — New tests for transform resolution (hint parsing, bounds inference, rotation propagation).

## 6  Mode-of-Operation

This section defines how the agent operates across iterative sessions on this project.

### 6.1  Session Structure
Each agent session follows this cycle:

```
[REFRESH]  → Load memory facts + read recent git log/diff
[REVIEW]   → Validate that prior work is correct and tests pass
[CAPTURE]  → Append new insights and change summaries to Plan_13_04_2026.md
[PLAN]     → Update the pending-work checklist with current priorities
[EXECUTE]  → Implement the highest-priority pending item(s)
[VALIDATE] → python -m pytest -q; parallel_validation (code review + CodeQL)
[COMMIT]   → report_progress with a descriptive commit message
[STORE]    → Update memory facts (store_memory) for next session
```

### 6.2  Priority Order
1. **Safety** — Never break print-safety invariants (Z≥0, smoothing cap, start/end preservation).
2. **Correctness** — All tests must pass before committing.
3. **Observability** — Warnings and logs for any fallback/default behaviour.
4. **Documentation** — Every user-visible change gets a doc update in the same commit.
5. **Performance** — Profile before optimising; gate changes behind env-var feature flags.

### 6.3  Code-Change Principles
- Make the **smallest safe change** that addresses the goal.
- New behaviour behind explicit conditions or env-var flags when risk is non-trivial.
- Every new function gets at least one unit test.
- Avoid side effects in helper functions — prefer pure transforms that return new state.

### 6.4  Memory Management
- After each session, store key facts via `store_memory` so they survive context resets.
- Outdated memory facts must be overwritten with current values.
- `Plan_13_04_2026.md` is the canonical persistent record; memories supplement it.

### 6.5  Documentation Policy
- Docs live in `docs/`; no free-standing markdown plans outside of this file.
- User-facing changes → update `docs/01-user-guides/`.
- Architecture changes → update `docs/02-technical-reference/`.
- Dev-only changes → update `docs/03-development/`.

### 6.6  Release Readiness Gates
- `python -m pytest -q` — all tests must pass.
- `parallel_validation` — no new CodeQL findings.
- CHANGELOG updated if the change is user-visible.
- `docs/` updated if behaviour changes.
